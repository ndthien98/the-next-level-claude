ANALYSIS — what to evaluate

A. Activity volume
   - Inbound count per project (from fleet.jsonl + per-project queue archaeology)
   - Commits + pushes + deploys per project
   - Specialist spawns per lead (estimate from tasks/*.output if visible)

B. Error patterns
   - Telegram send failures (tg-send-errors.log)
   - Skipped inbounds (inbound-skipped.log)
   - Tool failures in task transcripts (Bash exit codes, retries, timeouts)
   - Any audit-fleet.sh runs that flagged issues (fleet.jsonl)

C. Stall / wedge detection
   - Currently in-flight items that are >60 min old
   - Leads that haven't acted in N hours when their queue is non-empty
   - Background Monitor tasks that may have died silently

D. Performance opportunities
   - Lead doing work that should have been parallelized (look at Bash chains)
   - Specialist spawns that ran serially when parallel possible
   - Repeated tool calls on the same file (could have cached)
   - Long Bash commands that could be shorter
   - Large outputs from tools (could have been filtered)

E. Resource utilization
   - Disk usage trend (compare today's du to yesterday's if previous audit exists)
   - Log files growing without rotation
   - /tmp accumulation worth cleaning
   - Memory bank consolidation candidates (multiple files saying the same thing)

F. Hard-rule compliance
   - Hard Rule 1: any commits / pushes / PRs / Jira writes that lacked explicit
     owner approval visible in fleet.jsonl?
   - Hard Rule 2: any AI/persona/agent-id strings leaked into:
     - commit subjects or bodies (across all project repos)
     - file contents (workspace files)
     - doc files in plans/ / research/ / daily-audit/
     Grep recursively for: persona names, lead-<name>, anthropic, "claude code",
     "by claude", "generated by", llm, "compiled by .*(?!<owner-name>)"
   - Identity mismatch: any commits where author != per-project identity registry

G. Security signals
   - Any leaked tokens / secrets in workspace files (search for: BEGIN PRIVATE KEY,
     api[_-]?key, secret, password — but skip results inside .env files which are
     gitignored and OK)
   - Any 0.0.0.0 / public-binding changes
   - Any open Discord interactions endpoints (signature verify disabled)
   - Any unauthenticated production endpoints

H. Doc drift
   - CLAUDE.md / JOBS.md describe state that doesn't match disk
   - Plans marked complete that still have open TODOs
   - Identities.json projects out of sync with projects.json

I. Config consolidation + cross-layer consistency (KEY for optimization)
   With full read of global `~/.claude/`, fleet `.claude/`, and every per-project
   `.claude/`, look for:
   - **Rule conflicts**: same instruction said differently across global / fleet /
     project (e.g. "use yarn" vs "use pnpm" vs "use npm"). Flag the source files.
   - **Duplication**: same skill, agent, hook, or rule defined in multiple places
     (global vs project). Could one canonical copy replace the duplicates?
   - **Drift between layers**: project CLAUDE.md says X, but the actual sub-repo's
     own .claude/INSTRUCTION.md says Y — which one is the lead actually reading?
   - **Unused capabilities**: skills / agents / hooks present in config but never
     fired (cross-check against tool-call history). Candidates for pruning.
   - **Missing capabilities**: patterns in tool-call history that suggest a missing
     hook / skill / agent (e.g. lead re-typing the same multi-step Bash dozens of
     times — that pattern wants a helper script).
   - **Reasoning-effort settings**: is xhigh used where high or medium would
     suffice? (cross-check token-cost per task type)
   - **Memory bank overlap**: the same fact recorded in ~/.claude/memory/,
     fleet memory, project memory, AND a plan doc — propose canonical home.
