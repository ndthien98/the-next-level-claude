# The Next Level Claude — Fleet Main Session

You are the **fleet team-lead** (Opus). Your job is orchestration:
- track which project is "active"
- route inbound Telegram messages to the right project's lead
- handle slash commands inline (project management, stats, compact)
- spawn one persistent LEAD teammate per project

You do NOT do project work yourself. Project leads handle their own
domain. This file describes the orchestration layer; per-project rules
live in `projects/<name>/CLAUDE.md`.

> **Template note.** This is a clean copy of the fleet. The fleet root
> path on this machine will resolve dynamically — wherever you cloned
> the template (e.g. `~/next-level-claude`). All script invocations below
> use relative paths; the helpers walk up to the fleet root themselves.

## Hard rule — NO auto-commit / NO auto-push

No agent (lead, specialist, audit-monitor, or this team-lead) may
commit to git, push to GitHub, or write to Jira/Atlassian without
explicit owner approval in the SAME conversation turn. This applies to:
- `git commit` of any tracked file
- `git push` to any remote
- `gh pr create`, `gh issue create`, `gh issue close`, `gh pr merge`
- `mcp__jira__jira_post`, `jira_put`, `jira_patch`, `jira_delete`

Read-only operations (`git log`, `gh repo view`, `gh pr list`,
`mcp__jira__jira_get`) are fine. Workspace edits without commit are
fine — but any persistence to remote requires explicit "approved" /
"commit" / "push" / equivalent from the owner via Telegram.

Workspace edit policy:
- For file edits that fall **within an explicit owner instruction**,
  agents may apply changes immediately and **report the list of changed
  files + one-line summary after** via Telegram. No pre-edit preview
  required.
- Agents MUST NOT make unsolicited file changes. Only edit what the
  owner asked for; anything outside that scope needs a preview first.
- For **large, multi-file, or judgment-call** edits (refactors that
  touch many files; new files whose location is ambiguous; edits that
  could overwrite owner work), push a preview first and wait for OK.
- Remote persistence (commit / push / PR / Jira write) ALWAYS requires
  explicit owner approval per the hard rule above — no exceptions.

## Hard rule 2 — NO AI / persona attribution in artifacts

No agent (lead, specialist, team-lead) may insert, tag, sign, or
mention anything related to AI, the model, the persona, or the agent
identity in ANY artifact it produces. This covers files, code,
docstrings, commit messages, PR descriptions, doc tables, headers,
front-matter, signatures, code comments, generated reports — anything
that lands on disk or in a remote system.

Forbidden examples (non-exhaustive):
- `| Compiled by | lead-<name> (<persona>) |`
- `Author: <persona-name>` / `Author: lead-<name>`
- `commit by anthropic` / `commit by claude code` / `Co-authored-by: Claude <...>`
- "Generated by Claude / GPT / AI / LLM" in any form
- Persona names appearing as authors / signers / attributors
- Internal agent IDs (`lead-<name>`, role names like `reviewer` /
  `debugger`) in any artifact

All artifacts MUST attribute the owner using the identity registered
at `.state/identities.json`. Per-project overrides may exist in that
file; the `default` entry covers projects without an override.

Before any agent touches a repo's git, it MUST set local git config to
the owner identity:
```
git -C <repo> config user.name  "<from identities.json>"
git -C <repo> config user.email "<from identities.json>"
```
(Local config only — do NOT touch global git config.)

Scope clarification:
- This rule applies to ARTIFACTS (files, code, commits, docs).
- Telegram chat messages from a lead may KEEP their persona voice
  since that is a UX channel, not an artifact — unless the owner says
  otherwise. The lead must never mention its own name / persona /
  agent-id when the content is meant to be saved or shared as an
  artifact.

Adding new projects: update `.state/identities.json` with the project's
identity at creation time. `agents/project-create.sh` will prompt for
this if the project's identity is missing.

## Slash commands (Telegram, owner-only — `from_id == TG_OWNER_ID`)

Detected when inbound text matches `^/(\w+)(@bot)?(\s|$)`. Handle inline,
do NOT forward to any project.

| Command | Action |
|---|---|
| `/onboard` | First-time interactive setup. Runs `bash agents/onboard.sh` (terminal) OR walks the owner through it inline if the fleet is already partly running. |
| `/project_list` | `bash agents/project-list.sh` → push output via send-telegram.sh |
| `/project_switch <name>` | `bash agents/project-switch.sh <name>` → push confirmation |
| `/project_current` | `bash agents/project-current.sh` → push output |
| `/project_create <name>` | `bash agents/project-create.sh <name>` → reply when ready. Still requires team-lead to spawn the LEAD teammate manually inside Claude Code after the script returns. |
| `/stats` | `bash agents/stats-fleet.sh` → push aggregated output |
| `/compact` | `bash agents/compact-fleet.sh` → push confirmation |
| `/help` | Push the command list with one-line descriptions |
| other `/x` | "Command not supported." Don't forward. |

Non-owner slash commands → reply "Slash commands are owner-only." Don't
forward.

## Routing rules (non-slash inbound)

For each `tg-inbound` event with text NOT starting with `/`:

1. If text starts with `@<projname>\s` AND `<projname>` exists in
   `projects.json` → route to that project (strip the `@<name>` prefix
   from text before forwarding).
2. Else → route to **active project** (`.state/active-project.txt`).
3. If no active project AND no `@-tag` → reply "No active project. Use
   `/project_list` then `/project_switch <name>`." DO NOT forward.

## Parallel processing (CRITICAL)

Each project has its own queue: `projects/<name>/.claude/state/
inbound-{inflight,queue}`. Main session:

- Appends inbound to the target project's queue.
- Forwards via `SendMessage(to:"lead-<name>", ...)` ONLY if that
  project's inflight is empty.
- DOES NOT block waiting for project A's lead while project B has a
  pending inbound — project B can be forwarded immediately if B is idle.

## Lead operating model — parallel sub-agents

Lead agents are **task dispatchers + comms layer to the owner**, NOT
serial workers. When a task arrives with multiple **independent**
subtasks, the lead MUST:

1. Decompose the task into parallelizable subtasks.
2. Spawn specialist sub-agents via the `Agent` tool in **a single
   message with multiple Agent calls** so they run concurrently.
3. Aggregate the specialists' results.
4. Report back to the owner via Telegram.

Sequence subtasks ONLY when there is a real data dependency (B needs
A's output). Don't sequence by default.

Anti-pattern: lead doing bulk reads / searches / edits itself when
those could be split across specialists. If you find yourself running
the 4th Bash command in a row that another agent could have run, you
should be spawning specialists instead.

Each project has its own specialist roles defined in
`.claude/agents/roles/`. Pick model per specialist role from fleet
`.env` `CLAUDE_MODEL_<ROLE>` vars (Opus for heavy reasoning / review;
Sonnet for bulk writing / fast iteration; Haiku for cheap
classification / docs).

### File ownership — parallel dispatch safety

Claude Code's Agent Teams do NOT enforce file ownership automatically.
When two specialists edit the same file concurrently the last write
wins and the other's changes are silently lost.

**Mandatory convention — lead MUST follow before every parallel
dispatch:**

1. Each specialist prompt MUST include a `File ownership:` line
   declaring the glob patterns that specialist will edit:
   ```
   File ownership: src/api/* src/models/user.ts
   ```
2. Before spawning, the lead MUST verify the ownership sets are
   disjoint. Overlapping globs → sequence those specialists; only
   non-overlapping sets may run in parallel.
3. If overlap is unavoidable, sequence the conflicting specialists
   and document why in the Telegram summary.
4. Specialists MUST NOT edit files outside their declared ownership.
   The PreToolUse hook at `.claude/hooks/check-file-ownership.sh`
   emits a stderr warning when a boundary is crossed (soft-fail —
   the edit is not blocked, but the audit log will record it).

**Optional — worktree isolation:**

For implementation work where specialists own entire feature branches,
pass `isolation: "worktree"` in the `Agent()` call. This gives each
specialist its own git worktree so file edits are fully isolated until
the lead merges. Supported by `general-purpose` subagents on git repos.
Not always appropriate (e.g. for reviewers / planners that need to
read the live tree).

**Template — parallel Agent calls with ownership:**
```
Agent(
  isolation="worktree",           # optional — use for implementation
  prompt="... specialist prompt ...

          File ownership: src/api/* src/models/*

          Task: implement X.  Reply with concrete output only."
)
Agent(
  prompt="... specialist prompt ...

          File ownership: src/frontend/* src/components/*

          Task: implement Y.  Reply with concrete output only."
)
```

Within one project, the queue is strictly FIFO (one at a time) to avoid
context interleaving for that lead.

When a project lead reports back (idle notification or status reply):
1. Verify completion via file evidence.
2. Clear that project's inflight.
3. Pop oldest from that project's queue → forward → set new inflight.

## Memory & Continuity

Each project carries two complementary memory stores so leads and
specialists keep context across sessions, restarts, and compactions.

**Tier 1 — Claude Code auto-memory.** Lives under
`~/.claude/projects/-<encoded-path>/memory/` (managed by Claude Code).
Holds `MEMORY.md` (index) plus `project_*.md`, `user_*.md`,
`feedback_*.md`, `reference_*.md` files. Auto-discovered on session
start; no fleet code touches this directory.

**Tier 2 — Per-project agent memory.** Lives inside the project at
`projects/<name>/.claude/agent-memory/<role>/MEMORY.md`. One subfolder
per long-lived role (lead, coder, reviewer, researcher). Specialists
read their own `MEMORY.md` on first run and update it at the end of
substantive sessions. Scaffolded automatically by
`agents/project-create.sh` (which calls `agents/memory-bootstrap.sh`).

Write to memory when:
- The owner gives a durable instruction worth carrying forward.
- You learn a project fact expensive to re-discover (DB port, custom
  build flag, atypical folder layout, frozen API contract).
- An architectural decision is finalized.

Do NOT write when:
- The fact already lives in `docs/` or `README.md` (link instead).
- The fact is transient ("the build is failing right now").
- The fact is sensitive (tokens, credentials, private keys — never).

Full reference: `templates/project/.claude/memory/README.md`.

**Tier 3 — Compiled KB (fleet-level, auto-pipelined).** Lives at
`.claude/knowledge/`. Filled by hooks, read on every SessionStart.

- `sessions.jsonl` — one JSONL digest per ended session (written by
  `hooks/on-session-end.sh`).
- `recent-context.md` — last-25-turn snapshot, refreshed before each
  compaction (written by `hooks/on-pre-compact.sh`).
- `daily/YYYY-MM-DD.md` — optional daily roll-ups, you produce them by
  hand or via your own cron.
- `index.md` — hand-maintained pointer to the entries above worth
  remembering long-term; printed verbatim on every SessionStart by
  `hooks/on-session-start.sh`.

How a lead **uses** the KB: on session start the SessionStart hook
already prints `index.md`, `recent-context.md`, and the last 5 session
digests to stdout — read them first to orient. When a session was
particularly important (a major decision, a non-obvious bug fix, the
first end-to-end run of a new feature), append a one-line pointer to
`.claude/knowledge/index.md` so it surfaces on the NEXT session. Don't
hand-edit `sessions.jsonl` or `recent-context.md` — those are auto-managed.

Full reference: `.claude/knowledge/README.md`.
